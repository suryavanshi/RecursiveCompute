Recursive Compute Inference Fabric

Memory-rich decode for agentic LLMs.

RCIF is an LPDDR6-first hardware, simulator, and runtime roadmap for 500B to 5T parameter inference. It trades surplus training-style FLOPs for local capacity, sustained bandwidth, deterministic token scheduling, and explicit scale-up proof.

Design Center

Decode is a memory and locality problem.

Long-lived coding sessions repeat prompts, files, instructions, and tool output. RCIF therefore treats sustained local-memory delivery and keeping each token inside its locality domain as the first constraints, then sizes compute and links around them.

Buy local capacity

LPDDR6 targets active weights, KV pages, and descriptor buffers without making HBM packaging the default cost structure.

Size compute from bytes

The FP4 array follows the sustained memory roofline. Extra matrix silicon must earn its place in the simulator.

Keep the token local

Prefix affinity, virtual KV, SRAM staging, and page-aware DMA aim to avoid remote movement during the token step.

Budget every collective

Peer links carry activations, metadata, KV movement, and MoE traffic only when the per-token latency budget closes.

Architecture

RCIF splits prefill, decode, and locality domains.

The first implementation focuses on RC-Token, a memory-balanced decode ASIC. Existing GPUs or a simulator can handle prefill while an 8-chip locality domain proves KV-resident follow-up turns before rack-scale expansion.

Token Step

Decode becomes a hardware-scheduled graph.

Firmware prepares descriptors and handles faults, but the per-token critical path is meant to stay in hardware: queue, schedule, gather KV, stream attention, run a bandwidth-sized tensor array, fence only necessary collectives, complete.

Descriptor stream

Host submits a compiled token-step graph with request IDs, dependency masks, and operation descriptors.

Scheduler

The token scheduler issues work in deterministic order while tracking dependencies and QoS.

KV MMU + local DMA

Virtual KV pages translate through the TLB; gather/scatter engines move local-DRAM K/V tiles into SRAM.

Attention + tensor

Streaming attention consumes paged KV, then a sustained-bandwidth-sized tensor array produces the next layer output.

Collective + complete

Peer links exchange only scheduled TP/EP partials; the model must keep collective time within its TPOT budget.

Roadmap

Build the fabric from measurable slices.

The repo plan stages the hardware in a way that keeps every step testable: characterize workloads, freeze descriptors, prove protocols, then add compute engines against golden models.

Phase 0-1

Workloads and oracles

Use synthetic and real agentic traces to set targets, then maintain Python/PyTorch references for KV, attention, tensor math, and event-model bottleneck analysis.

  • 32K to 1M token prefix traces
  • 500B dense and 1T-5T MoE sizing
  • TTFT, TPOT, hit-rate, sustained local DRAM, and per-neighbor fabric targets
Phase 2-3

RTL protocols

Stabilize command formats, queues, status codes, KV page semantics, and DMA behavior before the engines become expensive to debug.

  • Descriptor package and register map
  • Fault and event queues
  • KV refcounting, prefetch, and page faults
Phase 4-5

Decode compute

Add paged streaming attention, then quantized tensor and activation units with tolerance-bounded comparison to golden models.

  • Online softmax over non-contiguous pages
  • GQA and MQA address mapping
  • FP8, FP4, INT8, and INT4 experiments
Phase 6+

Scale-out system

Move from one token engine to locality-aware groups only after bytes-per-token and collective-latency models close for each dense or MoE partition.

  • AllReduce, AllGather, ReduceScatter, and AllToAll
  • Tensor, expert, and limited pipeline parallelism
  • 8-chip locality domains and hot-layer replication

RL-Guided Optimization

Future tuning turns hardware counters into policy.

The long-term opportunity is to optimize the system as an adaptive inference environment. Simulator traces and hardware counters can train policies that choose routing, batching, KV tiering, precision, eviction, and locality placement under latency, memory, and peer-link constraints.

Observe

Measure the workload

Collect token traces, prefix hashes, KV hit/miss paths, local-DRAM efficiency, peer-link utilization, tail latency, power, and quality metrics.

Act

Choose system knobs

Adjust routing affinity, page size, channel allocation, batching windows, speculative decode, KV precision by tier, and eviction weights.

Reward

Score the policy

Reward accepted tokens per second while penalizing p95/p99 TPOT, TTFT, quality regressions, page faults, bandwidth spikes, and power cap violations.

Deploy

Constrain and verify

Ship only bounded policies with safe fallbacks, deterministic replay, tenant isolation, and regression tests over archived agentic traces.

maximize accepted_tokens_per_second subject to TTFT_p95 <= target, TPOT_p95 <= target, error_rate <= target, power <= cap, KV_hit_rate >= target, collective_share_of_TPOT <= target
