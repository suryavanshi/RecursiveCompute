Recursive Compute Inference Fabric

Custom inference systems for agentic LLM workloads.

RCIF is a hardware, simulator, and runtime roadmap for 500B to 5T parameter inference where repeated repository context, KV-cache locality, and deterministic decode latency matter more than training-style peak throughput.

Design Center

Agentic coding changes the bottleneck.

The repo treats long-lived coding sessions as the workload that should shape the silicon. The same prompts, files, instructions, and tool outputs recur, so the system makes KV reuse and low-jitter decode first-class architecture goals.

Large stable prefixes

System prompts, repo summaries, hot files, issue context, and tool results can stay resident as immutable prefix objects.

Latency-sensitive decode

Agents feel slow when token generation has poor p95 or p99 behavior, even if average throughput looks healthy.

Hardware-visible KV

KV pages get virtual addresses, tiers, format tags, refcounts, and routing metadata instead of hiding behind opaque software caches.

Deterministic orchestration

Compiled token-step descriptors and predictable scheduling reduce jitter and make verification more reproducible.

Architecture

RCIF splits prefill, decode, and KV locality.

The first implementation focuses on RC-Token, a decode-oriented ASIC role. Existing GPUs or a simulator can handle prefill while the custom fabric proves the value of KV-resident follow-up turns.

Token Step

Decode becomes a hardware-scheduled graph.

Firmware prepares descriptors and handles faults, but the per-token critical path is meant to stay in hardware: queue, schedule, gather KV, stream attention, run tensor work, fence collectives, complete.

Descriptor stream

Host submits a compiled token-step graph with request IDs, dependency masks, and operation descriptors.

Scheduler

The token scheduler issues work in deterministic order while tracking dependencies and QoS.

KV MMU + DMA

Virtual KV pages translate through the TLB; gather/scatter engines move hot K/V tiles into SRAM.

Attention + tensor

Streaming attention consumes paged KV, then tensor and MLP units produce the next layer output.

Collective + complete

Scale-up links exchange partials for TP/EP layouts before the completion queue reports progress.

Roadmap

Build the fabric from measurable slices.

The repo plan stages the hardware in a way that keeps every step testable: characterize workloads, freeze descriptors, prove protocols, then add compute engines against golden models.

Phase 0-1

Workloads and oracles

Use synthetic and real agentic traces to set targets, then maintain Python/PyTorch references for KV, attention, tensor math, and event-model bottleneck analysis.

  • 32K to 1M token prefix traces
  • 500B dense and 1T-5T MoE sizing
  • TTFT, TPOT, hit-rate, and bandwidth targets
Phase 2-3

RTL protocols

Stabilize command formats, queues, status codes, KV page semantics, and DMA behavior before the engines become expensive to debug.

  • Descriptor package and register map
  • Fault and event queues
  • KV refcounting, prefetch, and page faults
Phase 4-5

Decode compute

Add paged streaming attention, then quantized tensor and activation units with tolerance-bounded comparison to golden models.

  • Online softmax over non-contiguous pages
  • GQA and MQA address mapping
  • FP8, FP4, INT8, and INT4 experiments
Phase 6+

Scale-out system

Move from one token engine to topology-aware groups with collective hardware and runtime partitioning across dense and MoE models.

  • AllReduce, AllGather, ReduceScatter, AllToAll
  • Tensor, expert, and limited pipeline parallelism
  • Prefix locality domains across a rack

RL-Guided Optimization

Future tuning turns hardware counters into policy.

The long-term opportunity is to optimize the system as an adaptive inference environment. Simulator traces and hardware counters can train policies that choose routing, batching, KV tiering, precision, and eviction under latency and power constraints.

Observe

Measure the workload

Collect token traces, prefix hashes, KV hit/miss paths, DMA stalls, tail latency, power, and quality metrics from simulation and RTL counters.

Act

Choose system knobs

Adjust routing affinity, page size, continuous batching windows, speculative decode, KV precision by tier, and eviction weights.

Reward

Score the policy

Reward accepted tokens per second while penalizing p95/p99 TPOT, TTFT, quality regressions, page faults, bandwidth spikes, and power cap violations.

Deploy

Constrain and verify

Ship only bounded policies with safe fallbacks, deterministic replay, tenant isolation, and regression tests over archived agentic traces.

maximize accepted_tokens_per_second subject to TTFT_p95 <= target, TPOT_p95 <= target, error_rate <= target, power <= cap, KV_hit_rate >= target
